# Pass 10 — 제품 완성도 & 참조(jotso.net) 대비 격차 점검

> 점검일 2026-06-20 · 관점: 제품 완성도, 참조 대비 결정적 격차, 다음 우선순위(P0/P1/P2)
> 근거: 코드(`app/`, `components/`, `lib/`) + `작업계획서.md` + `DESIGN.md` + `PROGRESS.md` + `Page_text_export.txt`(실제 jotso.net 회사 상세 추출본).
> jotso.net 직접 fetch는 403(봇 차단). 따라서 참조 비교는 `Page_text_export.txt`(저장소에 보관된 실측 추출본)를 근거로 한다.

---

## 점검 범위

- 현재 구현된 라우트: `/`(검색), `/company/[id]`, `/monthly`, `/memorial`, `/awards`, `/game` + `/api/search`, `/api/nearby`.
- 데이터: `lib/data.ts` 인메모리 — 큐레이션 8개사(부록 A 실측) + 결정적 생성 500개사 = **508개사**. 실데이터(공공데이터 CSV/Supabase) 미적재.
- 비교 기준: `Page_text_export.txt`에 박제된 실제 jotso.net 회사 상세(주식회사 샤이닝라이언) 화면 구성.

> 주의(계획 vs 현실의 모순): `작업계획서.md`는 §0/§8.4에서 별이된좋소·시상식·게임·이달의좋소·월드컵·공유를 **명시적으로 버린다(범위 외)**고 선언했다. 그러나 실제 코드는 그 4개 코너 페이지를 모두 구현했다(`git log`: "코너 페이지 + 전역 네비"). 즉 **구현이 계획서를 추월/역행**했고, 두 문서가 현재 SSOT로서 충돌한다. 이 점검은 "jotso.net 클론으로서의 완성도"를 우선 기준으로 본다(사용자가 이번에 제시한 관점이 그쪽이므로).

---

## 발견 (격차, 심각도)

### [P0 / Critical] 1. 실데이터 부재 — 검색하면 더미만 나온다
- `lib/data.ts:326` `COMPANIES = [...SEEDS, ...genSeeds(500)]`. 508개 중 500개가 `genSeeds()`로 한글 어간 조합("한빛소프트", "미래테크" 등) + `mulberry32` 난수로 만든 **가짜 회사**다. 실존하지 않으며, 직원수/연봉/회전율도 난수.
- jotso.net의 핵심 가치 제안은 "**내 회사 이름을 치면 실제로 나온다**"이다. 현재 클론은 임의의 실회사명(예: "삼성전자", "네이버")을 쳐도 안 나오고, 큐레이션 8개사(샤이닝라이언/맑은소프트/아나패스/엠피테크 + 구로동 클러스터)만 의미 있는 결과다. 이것이 사용자 핵심 불만의 정체이며, **제품이 "진짜"가 되지 못하게 막는 단일 최대 격차**다.
- `작업계획서.md` M9(Supabase 적재 + Vercel 배포)가 미완. `PROGRESS.md`도 "M9: 실데이터 ETL 미완"으로 인정. 데이터 접근 함수(`searchCompanies`/`nearbyCompanies`/`getMonthlyStats`/`getCompany`)가 `lib/data.ts`에 격리돼 있어 교체 지점은 명확하나, **실제 ETL/적재 작업 자체가 0%**다.

### [P0 / Critical] 2. 회사 URL이 실사업자번호 기반이 아니다
- 참조는 `/company/340116023813` 형태의 숫자 ID(`Page_text_export.txt`의 위험도 사다리 링크들: `/company/012895039723`, `/company/376895941833` 등 12자리 숫자).
- 클론은 큐레이션사가 `shininglion`/`malgunsoft` 같은 **영문 슬러그**(`lib/data.ts:50~`), 생성사만 12자리 숫자(`lib/data.ts:297` `String(100000000000 + ...)`)다. 혼재돼 일관성이 없고, `작업계획서.md` §6의 "`(사업장명+6자리사번+법정동코드)` 해시 base36 stable ID" 규칙과도 다르다(현재는 해시도 base36도 아님).
- 실데이터가 들어오면 stable ID 생성 규칙을 §6대로 확정해야 하므로, 이는 P0(실데이터)와 묶여 있다.

### [P0 / Critical] 3. 검색이 사업자번호로 안 된다 (카피-동작 불일치)
- `SearchBox.tsx:71` placeholder = "회사명 **또는 사업자번호** 검색". 그러나 `searchCompanies()`(`lib/data.ts:387`)는 `biz_name`만 `includes`로 매칭한다. `biz_no6`는 검색 대상이 아니다.
- 사용자에게 "사업자번호로 검색된다"고 약속하고 실제론 안 되는 **거짓 카피**. placeholder를 고치거나(저비용) 검색을 사번까지 확장해야 한다.

### [P1 / High] 4. 회사 상세에 참조의 시그니처 블록들이 통째로 누락
`Page_text_export.txt`(실제 화면)에 있으나 클론(`app/company/[id]/page.tsx`)에 **없는** 블록:
- **"그때 vs 지금"** — "내가 입사한 해" 선택 → "6명→49명 ▲717%" 증감 서사. jotso의 바이럴 핵심 위트. 완전 부재.
- **위험도 사다리(더좋소/덜좋소)** — 같은 위험도 근처 회사를 위/아래로 나열. 부재.
- **"OO구 좋소 순위"** — "종로구 8위 / 2,658곳 중 · 상위 1%". 지역 내 랭킹. 부재.
- **"OO에서 갈아탈 만한 곳"**(위험도 낮은 순) — 클론의 `nearbyCompanies`는 **연봉순 1종**만 제공(`lib/data.ts:418`). 참조는 "갈아탈 곳(저위험)"과 "같은 일인데 더 받는 곳(고연봉·배수표기)"을 **분리**해 보여준다. 클론은 후자 일부만, 배수("7.9배") 표기 없이.
- **공유 카드(OG 이미지) / 카카오톡 공유 버튼** — 참조엔 `/og/...` 카드 + "카카오톡/복사" 버튼이 헤더·캡슐 양쪽에. 클론은 공유 기능 전무.
- **월드컵 링크** — "OO구 좋소 월드컵". 부재(계획서상 버린 항목이긴 함).

### [P1 / High] 5. 데이터 출처/갱신월 표기가 약하다 (신뢰 요소)
- 푸터에 "공공데이터(국민연금) 기반 추정치"는 있으나(`app/company/[id]/page.tsx:150`), **자료생성년월(기준월)이 화면에 안 보인다**. `Company.last_ym`("2026-04")이 데이터에 있는데(`lib/data.ts:217`) 어디에도 렌더하지 않는다.
- 참조는 차트마다 "2024-08 ~ 2026-04" 기간을 명시. 클론 차트는 기간 축은 있으나 "데이터 기준월/갱신월" 명시가 없어, 실데이터 적재 시 신뢰도(언제 기준 데이터인가)를 전달하지 못한다.

### [P1 / High] 6. 코너 페이지의 데이터가 전부 가짜라 의미가 없다
- `/monthly`(이달의 좋소 TOP20), `/memorial`(휴폐업), `/awards`(부문 1위), `/game`은 모두 `COMPANIES`(=500 더미)에서 뽑는다. 즉 "이달의 좋소 1위 = 한빛테크(난수)" 식으로, **실존하지 않는 회사가 시상대에 오른다.** 페이지 골격은 완성됐으나 데이터가 가짜라 P0(실데이터) 해결 전에는 콘텐츠 가치 0.
- `/memorial`은 `isClosed: rnd() < 0.03`(`lib/data.ts:317`)로 무작위 3%를 폐업 처리 — 완전 허구.

### [P2 / Medium] 7. 점수 엔진은 8개사에만 검산됨, 나머지는 미검증 계수
- `score.ts`의 계수(회전율 `(t-20)*0.22`, 연봉 `35*(1-ratio)`)는 `작업계획서.md` §9의 1차 보정값. 큐레이션 8개사는 `forceScore`로 실측값 하드코딩(`lib/data.ts:66` 등)이라 항상 맞지만, **생성 500개사·향후 실데이터는 미검증 엔진을 그대로 탄다.** 실데이터 적재 시 원본 점수와 대조 검산(M3 DoD "샤이닝라이언≈78, 아나패스≈0")을 실데이터로 재수행해야 한다.

### [P2 / Medium] 8. 톤/카피 일관성 — 위트는 살아있으나 산발적
- 코멘트 톤("도망치세요. 이건 훈련 상황이 아닙니다.")은 jotso 감성을 잘 잡았다(`lib/data.ts:64`). 다만 생성사는 `commentForScore()` 점수구간 템플릿 6종으로 단조롭다(`lib/data.ts:225`). 참조의 "그때 vs 지금" 같은 **데이터에서 자동 생성되는 서사**가 없어, 위트가 헤더 한 줄에 갇혀 있다.
- `/awards` 부문명("회전문 대상", "박봉 대상")은 톤이 좋음. 일관성 자체는 양호.

### [P2 / Low] 9. 즐겨찾기 / 비교 / 최근 검색 등 부가 UX 전무
- 즐겨찾기, 회사 2곳 나란히 비교, 최근 본 회사 같은 재방문 유도 기능이 없다. 참조에도 핵심은 아니나, 제품 완성도 측면의 빈칸. (실데이터 이후 P2.)

## 우선순위 로드맵 (P0/P1/P2)

### P0 — 이게 돼야 "진짜 제품"이 된다 (실데이터 축)
1. **공공데이터 ETL 실행 (M9 절반)** — `국민연금공단_국민연금 가입 사업장 내역` 최신월 CSV 1개를 받아, `작업계획서.md` §7 순서대로 파싱→정규화→stable ID(§6 해시 base36)→`monthly_stats`/`companies`/`industry_median` 산출. 우선 **로컬 SQLite 또는 JSON**으로라도 적재해 `genSeeds(500)`를 **실데이터로 치환**한다(Supabase는 그다음). 교체 지점은 `lib/data.ts`의 4개 접근 함수뿐이라 구조는 준비됨.
2. **stable ID 규칙 확정 + URL 통일** — 큐레이션 영문 슬러그를 §6 규칙 ID로 이관하고, `/company/[id]`를 전부 동일 체계(숫자/해시)로. 실데이터 적재와 동시에 처리.
3. **검색 카피-동작 일치(즉시·저비용)** — `searchCompanies`에 `biz_no6` 부분일치 추가하거나 placeholder에서 "사업자번호" 문구 제거. P0 중 가장 싸므로 지금 당장.
4. **데이터 기준월 노출** — `last_ym`을 회사 상세 헤더/푸터와 코너 페이지 상단에 "데이터 기준: 2026-04" 형태로 표기. 신뢰의 최소 요건.

> P0 완료 정의(DoD): 실재하는 회사명("삼성전자" 등 흔한 사업장)을 검색해 결과가 나오고, 그 상세의 직원수/회전율이 CSV 원본과 손계산으로 일치. 코너 페이지에 실존 회사가 오른다.

### P1 — jotso다움을 채운다 (참조 시그니처 블록)
5. **"그때 vs 지금" 입사년도 서사** — `monthly_stats` 시계열로 "입사한 해 선택 → N명→M명 ▲X%" 카드. jotso의 바이럴 위트 핵심. 데이터가 이미 있으니 UI만.
6. **추천 2분화** — `nearbyCompanies`를 (a) "갈아탈 만한 곳"(같은 지역 `risk_score` 낮은 순) + (b) "같은 일인데 더 받는 곳"(같은 업종 연봉순 + **자사 대비 배수** 표기)로 분리. `작업계획서.md` §8.3 6번에 이미 명시된 "(보조) 갈아탈 만한 곳" 미구현분.
7. **지역 순위 + 위험도 사다리** — "OO구 N위/총M곳·상위P%", 같은 위험도 ±근방 회사 리스트. 둘 다 `COMPANIES` 정렬로 산출 가능(인메모리에서도 즉시).
8. **공유(OG 이미지 + 복사 버튼)** — Next `ImageResponse`(`/og/[id]`)로 위험도 카드 생성 + URL 복사 버튼. 카카오 SDK는 키 필요하니 우선 "링크 복사"부터. 참조의 핵심 확산 장치.

### P2 — 완성도 마감
9. 점수 엔진 실데이터 재검산(원본 대조), 계수 미세조정.
10. 생성 코멘트를 데이터 파생 서사로 풍부화(템플릿 6종 → 지표 기반 동적 문구).
11. 즐겨찾기 / 최근 본 회사 / 2곳 비교.
12. 계획서 vs 구현 SSOT 정리: `작업계획서.md` §0/§8.4의 "코너 페이지 버림" 선언과 실제 구현이 충돌 → 문서를 현실(코너 페이지 유지)에 맞게 갱신하거나 명시적으로 "범위 확장됨" 기록.

## 반성점

- **데이터가 제품의 전부인데, 가장 어려운 데이터(P0)를 가장 마지막으로 미뤘다.** "목업 우선 전략"(`PROGRESS.md`)은 화면을 빨리 띄우는 데는 옳았으나, 500개 더미가 "완성된 것처럼" 보이게 만들어 **미완을 가린다.** 검색·코너·시상식 모두 화면은 그럴듯한데 알맹이가 가짜라, 진척률을 과대평가하기 쉽다. 점검자는 화면 스크린샷이 아니라 "검색어가 실회사냐"로 완성도를 재야 한다.
- **계획서와 구현이 정면 충돌하는데 아무도 정리하지 않았다.** `작업계획서.md`는 코너 페이지를 명시적으로 버렸는데 코드는 다 만들었다. SSOT가 둘로 갈라진 채 방치돼, "무엇이 계획 대비 미완인가"라는 질문 자체가 모호해졌다. 범위 변경은 결정과 동시에 문서에 반영했어야 한다.
- **참조 추출본(`Page_text_export.txt`)을 갖고 있으면서 그 안의 절반(그때vs지금·사다리·지역순위·공유)을 구현 목록에서 누락했다.** 가장 신뢰할 수 있는 스펙이 손에 있었는데, 구현은 "핵심 지표 4종 + 차트 3종 + 주변추천"에서 멈췄다. 참조 화면을 블록 단위 체크리스트로 만들어 대조했다면 격차가 더 일찍 드러났을 것이다.
- 솔직히, 현재 상태는 **"jotso.net처럼 보이는 껍데기"**다. 디자인 시스템(`DESIGN.md`) 충실도와 코드 구조(접근 함수 격리)는 우수하나, 제품으로서는 P0 실데이터 한 방이 빠져 "체험판"에 머문다.
