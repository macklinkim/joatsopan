# Round2-03 — 검색 품질 / 적합성 점검

대상: `jotsopan` 검색 경로 — `lib/data.ts`의 `searchCompanies` (L130-146), `app/api/search/route.ts`, `components/SearchBox.tsx`. 실데이터 552,878개 활성 사업장(`data/companies.json`).
점검 일자: 2026-06-20.

## 점검 범위 / 방법

- 코드 Read: `searchCompanies` 매칭/정렬/캡 로직, API 라우트, SearchBox 디바운스·키보드·에러 처리.
- 동작 검증: dev 서버(`http://localhost:3000`) 가동 확인(200). `/api/search?q=` 를 실제 질의.
  - 주의: Windows Git Bash + curl 로 한글 쿼리를 보내면 바이트가 깨져 전부 `{"results":[]}` 가 나온다(셸 인코딩 문제이지 앱 버그 아님). 그래서 검증은 **Node `fetch`(UTF-8 정상)** 로 다시 수행했고, ASCII 쿼리(`SAMSUNG`, `123`)는 curl 로도 확인했다.
- 데이터 직접 분석: `data/companies.json` 을 Node 로 로드해 매칭 수·정렬·사업자번호 분포를 측정(아래 수치는 실측).

## 발견 (심각도 / 근거)

### F-1. [양호 — 검증됨] 600 캡이 대형사를 누락시키지 않는다
- 우려: 흔한 단어가 600 후보 캡(L141 `if (hits.length > 600) break;`)에 먼저 걸려, 캡 이후에 있는 대형사가 상위10에서 빠질 수 있는가?
- 근거(실측): `A.name`/`A.members` 배열은 **가입자수 내림차순으로 완벽 정렬**되어 있다 — 인접쌍 552,877개 중 역전 0건. members[0..5] = [125614, 66559, 42752, 35312, 34474, 25425].
- 따라서 스캔(L138 `for i=0..NA`)은 큰 회사부터 만난다. 최악 케이스 "주식회사"는 총 220,728건 매칭이지만 **601번째 매칭 시점의 회사는 이미 380명**(서울=45명, 병원=137명, "(주)"=659명, "주"=982명). 상위10은 항상 첫 600 히트 안에 들어오므로 캡은 **결과 품질에 무해**하고 풀스캔만 절약한다.
- 단, 이 보장은 "배열이 members desc 정렬"이라는 ETL 산출물의 불변식에 전적으로 의존한다. 정렬 가정이 깨지면(예: 다른 정렬로 재생성) 캡이 곧바로 정확도 버그가 된다. 코드에 이 의존성을 명시한 주석/단언이 없다(개선 P-1).

### F-2. [중간] "사업자번호"는 6자리 = 회사당 고유 식별자가 아님 → 번호 검색 오탐 다수
- 근거(실측): `bizNo` 는 552,878건 전부 **길이 6**. 그러나 **distinct 값은 32,241개뿐**(중복 520,637건). 즉 한 6자리 코드를 평균 ~17개 회사가 공유한다. 이는 실제 사업자등록번호(10자리)가 아니라 앞자리/부분 코드로 보인다.
- 영향:
  - L136 `byDigit = digits.length >= 3` 로 3자리 이상 숫자가 들어오면 번호 매칭이 켜진다. 6자리를 정확히 입력해도 **수십 개 회사가 동률로 매칭**되어 사용자가 특정 회사를 번호로 핀포인트할 수 없다. "사업자번호 검색"이라는 placeholder 약속(SearchBox L71)과 어긋난다.
  - `1234567`(7자리)는 결과 0 — 6자리 데이터에 7연속 숫자가 없어서다. 사용자가 진짜 10자리 사업자번호를 넣으면 거의 항상 무결과가 된다.
- 오탐 측정(q="123"): 이름에 "123"이 들어간 회사 33건, 사업자번호에 "123" 포함 3,487건. 즉 숫자 쿼리는 사실상 번호 검색으로 동작하지만 "123스텝업병원", "주식회사123푸드", "일이삼내과(123내과)" 처럼 **이름의 숫자**도 같이 잡힌다(L139 OR 조건). 의도가 회사명인지 번호인지 구분 없이 합쳐진다.

### F-3. [중간] 적합도(랭킹)가 "가입자수"뿐 — 부분일치/접두/완전일치 가중 없음
- 근거: L139는 `includes` 부분일치 불리언일 뿐이고, L144 정렬은 `A.members[b] - A.members[a]` 단일 키. 매칭 위치(접두 vs 중간), 완전일치 여부, 매칭 길이 비율을 전혀 보지 않는다.
- 영향: "소프트"로 검색하면 이름 어디에든 "소프트"가 들어간 가장 큰 회사가 1위(한국마이크로소프트). 사용자가 상호를 정확히 떠올려 입력해도, 완전일치/접두일치 회사가 더 큰 무관한 회사 아래로 밀릴 수 있다. 자동완성 UX에서는 보통 **접두일치 > 단어경계 일치 > 부분일치** 가중을 주는데 그 신호가 없다.
- 완화: members 정렬은 "유명한 곳이 위"라는 직관과 대체로 맞아 체감 품질은 나쁘지 않다(검증표 참고). 하지만 "정확히 그 이름"을 노린 질의에는 약하다.

### F-4. [낮음] 한글 정규화(NFC/NFD)·전각/공백 처리 없음
- 근거: L133 `q.trim().toLowerCase()` 만 한다. Unicode 정규화(`normalize("NFC")`)가 없다. 데이터는 NFC 로 보이고(삼성=ec82bc..., includes 정상) 질의도 보통 NFC 라 현재는 문제가 표면화되지 않는다. 그러나 macOS 파일명/일부 IME 가 만드는 NFD 자모분리 입력이 들어오면 `includes` 가 실패한다.
- 전각/특수괄호: 데이터에 `（주）`, `．`, `&` 등 전각·특수문자가 섞여 있다(예: "주식회사삼성엠에이치（SAMSUNGMHCO．Ltd）"). 사용자가 반각 "(주)"로 쳐도 전각 "（주）" 회사는 매칭 안 된다. 공백도 그대로라 "주식회사 카카오" vs "주식회사카카오" 처럼 공백 유무로 갈린다(데이터에 둘 다 존재).
- 대소문자: `toLowerCase` 양쪽 적용되어 ASCII 대소문자는 정상(`SAMSUNG` 검증 통과). nameLc 는 모듈 로드 시 1회 전처리(L131)로 매 질의 비용 없음 — 잘한 부분.

### F-5. [낮음] 동점 정렬 안정성 — 사실상 안정적이나 명시 부재
- 근거: L144 비교자는 members 차이만 반환. members 동률이면 비교자가 0 → V8 `Array.prototype.sort`는 안정 정렬(ES2019+)이라 **원래 인덱스 순서(= members desc 내 등장순)** 가 유지된다. 따라서 같은 쿼리는 결정적·재현 가능.
- 리스크: 안정성이 스펙에 의존(엔진 보장)하고, 동률 타이브레이커(이름 사전순 등)가 명시돼 있지 않아 "왜 이 순서?"가 불투명하다. 기능 버그는 아님.

### F-6. [낮음] 무결과 처리 — 정상
- API: 무매칭 시 `{"results":[]}` (route L7-14, 항상 200). 빈 쿼리/공백 쿼리는 L134 `if(!norm) return []`. `q` 파라미터 누락도 `?? ""` 로 안전(route L6).
- UI: SearchBox L95-99 가 `open && q.trim() && results.length===0` 일 때 "검색 결과가 없습니다." 표시. 빈 입력일 때는 드롭다운을 안 띄운다. 무결과 UX 양호.

### F-7. [중간] SearchBox 디바운스/경합(race)·에러 처리
- 디바운스: L16 `setTimeout(...,180)` + L31 cleanup 으로 입력당 타이머 리셋. 기본 동작 정상.
- **경합 미처리(stale response)**: L22 `fetch` 응답이 도착 순서대로 `setResults`(L24) 된다. 디바운스가 180ms 라 빠른 타이핑은 대부분 직전 타이머를 취소하지만, 디바운스 통과 후 **연속 발사된 두 요청의 응답이 역순으로 도착**하면 오래된 결과가 최신 입력 위에 덮인다. AbortController 나 요청 시퀀스(요청 id 비교)가 없다. 캡 600·풀스캔으로 응답 시간이 들쭉날쭉할 수 있는 실데이터에선 드물게 표면화 가능.
- 에러 처리: L27 `catch { setResults([]) }` 로 throw 는 잡지만 — (a) `res.ok` 검사가 없어 500/4xx 도 `res.json()` 을 시도하고(파싱 실패 시에만 catch), (b) 에러 시 `setOpen` 을 안 만져서 직전 드롭다운 상태가 남을 수 있다. 사용자에게 에러 표시는 없음(무결과와 구분 불가).
- 마이너: L69 `onFocus` 는 `results.length` 일 때만 open. cleanup 은 적절. 외부 클릭 닫기(L34-40) 정상.

## 개선 제안 (우선순위)

- **P-1 [F-1]** `searchCompanies` 상단에 "A 배열은 members desc 정렬 전제, 600 캡은 그 위에서만 정확" 주석 + 가능하면 ETL 산출 시 정렬 단언 추가. 캡 의존성을 명문화.
- **P-2 [F-2]** "사업자번호 검색"을 재정의: 6자리는 고유키가 아니므로 placeholder 문구를 "상호명 검색"으로 바꾸거나, 진짜 10자리 번호를 데이터/검색에 도입. 최소한 숫자 쿼리를 이름-숫자와 번호로 분리하거나, 번호 매칭 시 동률 다수를 사용자에게 알리기.
- **P-3 [F-3]** 랭킹에 접두/완전일치 가중 추가: `score = (완전일치?3:접두?2:1) * f(members)` 형태. members 만으로 정렬하던 것을 1차 적합도, 2차 규모로.
- **P-4 [F-4]** 질의·데이터 양쪽에 `normalize("NFC")` 적용, 전각→반각·괄호 정규화, 공백 제거 비교 옵션. nameLc 전처리 시 함께 정규화.
- **P-5 [F-7]** SearchBox 에 AbortController(이전 요청 abort) 또는 요청 시퀀스 가드 추가, `res.ok` 검사, 에러 상태 UI 분리.
- **P-6 [F-5]** 동률 타이브레이커(이름 길이/사전순) 명시로 순서 의도 고정.

## 종합

검색의 **핵심 기능 품질은 양호**하다 — 한글 부분일치 정상, 대형사 상위 노출 정상, 600 캡은 members-desc 정렬 덕분에 결과를 누락시키지 않음(실측 확인), 무결과·빈쿼리 처리 견고, nameLc 전처리로 매 질의 비용 절약. 한글 쿼리 무결과로 보이던 것은 curl 셸 인코딩 artifact 였고 앱 버그가 아니다. **개선 여지**는 적합도 정교화(접두/완전일치 가중 부재), 사업자번호 6자리의 비고유성(번호 핀포인트 불가·오탐), 정규화(전각/NFD/공백), SearchBox 응답 경합·에러 처리에 있다.
