> 근거: `Page_text_export.txt`(실제 jotso.net 회사 상세 추출본·SSOT) + 코드(`app/`, `components/`, `lib/`) + `작업계획서.md` + 직전 `docs/review/pass-10-product-completeness.md`.

## 전제 변화 — P0(실데이터)는 해소됐다

직전 pass-10의 최대 격차였던 "검색하면 더미만 나온다(508개 중 500개 가짜)"는 **해소됐다.**

- `lib/data.ts:18` 이 `data/companies.json`(전국 국민연금 가입 사업장 **552k 전수**)를 컬럼형+인터닝으로 적재한다. `genSeeds(500)` 난수 더미는 사라졌다.
- `searchCompanies`(`lib/data.ts:132`)는 활성 전수를 스캔하며 **회사명 + 사업자번호 6자리** 양쪽으로 매칭(`nameLc[i].includes` ‖ `bizNo[i].includes`) — pass-10이 지적한 "사업자번호 검색 거짓 카피"도 해결됨.
- stable ID는 §6 규칙(`name+bizNo+bdong` 해시 base36 + 충돌 접미사, `lib/data.ts:44~58`)으로 통일 — 영문 슬러그 혼재 격차도 해결.
- 데이터 기준월(`DATA_YM`)이 홈(`app/page.tsx:9`)에 노출됨.
- 코너 4종(이달의좋소/별이된좋소/시상식/게임)이 **실데이터에서 산출**(`topRiskCompanies`/`closedCompanies`/`awards`/`gamePool`). pass-10이 지적한 "허구 시상대"도 실존 회사로 채워짐.

즉 제품은 pass-10의 "체험판" 단계를 졸업했다. **남은 격차는 회사 상세 화면의 '서사·확산' 블록**에 집중돼 있다. 이것이 원본 바이럴의 핵심이다.

`Page_text_export.txt`(원본 화면)와 `app/company/[id]/page.tsx`(현재)를 블록 단위로 대조.

### [P1] 1. "그때 vs 지금" 입사년도 서사 — 원본 바이럴의 심장
원본은 "내가 입사한 해(2024/2025/2026)" 선택 → "6명→49명 ▲717%", "당신이 들어온 2024년보다 **43명** 늘었어요"라는 **개인화 서사**를 만든다(L63~73). 이게 캡처돼 SNS로 퍼지는 핵심 장치다. 시계열 데이터(`getMonthlyStats`)는 이미 있으므로 **UI/계산만** 추가하면 된다. 완전 부재.

### [P1] 2. 위험도 사다리(더좋소/덜좋소)
같은 위험도 ±근방 회사를 위/아래로 나열(L75~91) — "내 회사보다 더한 곳도 있다"는 상대화 위트. 현재 `topRiskCompanies`가 전역 정렬 캐시를 갖고 있어, 점수 근방 슬라이스로 즉시 구현 가능.

### [P1] 3. 지역 순위 "OO구 N위 / M곳 중 · 상위 P%"
원본 L93~95. 같은 `sigungu` 내 위험도 정렬에서 자기 순위/총수/백분위 산출. 인메모리 정렬로 즉시 가능(다만 552k 전수 매 요청 스캔이므로 시군구별 인덱스 1회 캐싱 권장 — 성능 주의).

### [P1] 4. 추천 2분화 + 배수 표기
현재 `nearbyCompanies`(`lib/data.ts:151`)는 **연봉순 1종**만(`page.tsx:141~147`). 원본은 (a) "갈아탈 만한 곳" = 같은 지역 **위험도 낮은 순**(L97~105), (b) "같은 일인데 더 받는 곳" = 같은 업종 연봉순 + **자사 대비 배수("7.9배")**(L107~115)로 **분리**. 데이터 접근은 동일, 정렬 키와 배수 계산만 추가.

### [P1] 5. 공유(OG 이미지 + 복사/카카오)
원본은 헤더·캡슐 양쪽에 `/og/...` 카드 + 카카오/복사 버튼(L17,25,65,73). 현재 `app/og/` 라우트 없음, 공유 버튼 전무. **확산 장치가 0** — "원본 핵심기능 이상" 목표에서 가장 큰 미달. Next `ImageResponse`로 위험도 카드 생성 + 링크 복사부터(카카오 SDK는 JS 키 필요).

### [P2] 6. 점수 엔진 실데이터 미검증
`score.ts` 계수(회전율 `(t-20)*0.22`, 연봉 `35*(1-ratio)`)는 8개 샘플 1차 보정값. 552k 실데이터에 그대로 적용 중이나, ETL 단계에서 원본 점수와 대조 검산한 기록은 없다(`A.score`는 ETL 산출이지만 원본 일치 검증 미기재).

### [P2] 7. 성능 — 전수 스캔 패턴
`searchCompanies`/`nearbyCompanies`/`awards`는 매 호출 552k 선형 스캔. 검색은 600개 캡으로 방어하지만, 지역 순위·사다리를 추가하면 요청당 다중 전수 스캔이 누적된다. 시군구/업종/점수 버킷 **사전 인덱스**(모듈 로드 시 1회) 필요.

## 우선순위 로드맵 (P0/P1/P2)

### P0 — 이미 충족 (유지·검증만)
- 실데이터 552k 적재 ✅ / 사번 검색 ✅ / stable ID 통일 ✅ / 기준월 노출 ✅ / 코너 실데이터화 ✅
- **유지 과제**: 회사 상세에도 `last_ym` 노출(현재 홈에만), 푸터 "데이터 기준 {DATA_YM}" 한 줄 추가(저비용).

### P1 — 원본 시그니처 블록 채우기 (jotso다움 = 바이럴)
1. **"그때 vs 지금" 카드** — `getMonthlyStats`에서 연도 시작값↔현재값 증감률·증감수 계산, 연도 토글 UI. (서버 계산 + 클라 토글 컴포넌트)
2. **추천 2분화 + 배수** — `nearbyCompanies`를 두 모드로: `salary`(현행) + `risk`(저위험순). "더 받는 곳" 카드에 `salary/자사salary` 배수 표기.
3. **위험도 사다리** — `topRiskCompanies` 캐시(또는 점수 정렬 배열)에서 자기 위치 ±3 슬라이스 → 더좋소/덜좋소.
4. **지역 순위 배지** — 시군구별 위험도 정렬 인덱스 1회 캐싱 → "N위/M곳·상위P%". (성능 위해 인덱스 동시 도입)
5. **공유**: `app/og/[id]/route.tsx`(`ImageResponse`) 위험도 카드 + 상세 헤더에 "링크 복사" 버튼 + `generateMetadata`로 OG 메타. 카카오는 키 확보 후.

> P1 DoD: 샤이닝라이언 상당 회사 상세에서 "그때 vs 지금", 더좋소/덜좋소, 지역순위, 더받는곳(배수), 공유 카드가 원본 화면 구성과 1:1 대응.

### P2 — 마감·신뢰
6. ETL 단계 점수 실데이터 재검산(샘플 N개 손계산 대조), 계수 미세조정 기록.
7. 사전 인덱스 도입(시군구·업종·점수 버킷)으로 다중 전수 스캔 제거.
8. 월드컵(`/worldcup?region=`) — 지역 토너먼트(코너 확장).

## 초과달성 로드맵 ('원본 이상' 차별화)

원본은 "한 회사를 재밌게 박제"하는 데 강하지만, **탐색·비교·통계는 약하다.** 552k 전수를 이미 메모리에 쥐고 있으므로(원본도 같은 공공데이터 기반) 아래는 우리만의 우위가 된다.

### T1 — 데이터 우위로 즉시 차별화 (구현 쉬움·임팩트 큼)
- **필터 검색** `/search?sido=&industry=&minMembers=&maxRisk=` — "서울 + IT + 위험도 30 이하" 같은 조건 탐색. 원본엔 없음. 인메모리 필터로 가능(인덱스 전제).
- **업종 비교 대시보드** `/industry/[code]` — 업종 중앙값·연봉 분포·위험도 분포·상하위 회사. `industry_median`이 이미 있으니 확장.
- **연봉 백분위** — 회사 상세에 "이 회사는 전국 동종업계 상위 P%" 한 줄. 업종별 연봉 정렬 인덱스로 산출. "업종 중앙값의 16%"보다 한 단계 위 정보.

### T2 — 재방문 유도 (제품화)
- **즐겨찾기 / 최근 본 회사** — localStorage 기반. 가입 없이도 "내 관심 회사" 추적.
- **회사 2곳 나란히 비교** `/compare?a=&b=` — 지표 4종·차트를 좌우로. "지금 회사 vs 이직 후보" 의사결정 도구. 이직 맥락(원본 "갈아탈 곳")의 자연스러운 상위 기능.

### T3 — 통계·인사이트 (콘텐츠 자산화)
- **전국 통계 대시보드** `/stats` — 시도별 평균 위험도, 업종별 회전율 순위, 월별 폐업 추이. 552k 집계는 ETL에서 사전 산출하면 즉시 렌더.
- **트렌드 코너** — "이번 달 가장 많이 사람이 나간 업종" 등 코너의 데이터 저널리즘화.

> 초과달성 우선순위 제언: **T1-필터검색 → T1-연봉백분위 → T2-비교**. 셋 다 (a)이미 가진 552k 데이터로 가능, (b)원본에 없음, (c)이직 의사결정이라는 사용자 실제 목적에 직결. 단 모두 **P2-7 사전 인덱스**가 선행 조건(전수 스캔 누적 방지).

## 반성점 / 근거 요약
- 직전 pass-10은 "실데이터 부재"를 P0 최대 격차로 봤고 그게 해소됐다 — 점검은 항상 코드 현재 상태(`data/companies.json` 적재 확인)부터 재확인해야 하며, 과거 리뷰를 SSOT로 신뢰하면 오판한다.
- 남은 격차는 전부 회사 상세의 "서사+확산" 블록(그때vs지금/사다리/지역순위/추천2분화/공유)에 모여 있다 — 데이터·접근 함수는 이미 있고 **UI·정렬·집계만** 빠졌으므로 P1은 저위험·고효과다.
- '원본 이상'의 진짜 레버는 552k 전수라는 데이터 자산이다. 원본이 못 하는 **탐색·비교·통계**로 가되, 그 전에 전수 스캔을 인덱스로 정리하지 않으면 기능 추가마다 응답 지연이 누적된다(성능이 차별화의 전제).
