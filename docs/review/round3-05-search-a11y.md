# Round3-05 — 검색 품질 / 정규화 / combobox 동작 점검

대상: `lib/data.ts` `searchCompanies` (L149-164), `app/api/search/route.ts`, `components/SearchBox.tsx`. 실데이터 552,693개 활성 사업장(`data/companies.json`, ym=2026-04).
점검 일자: 2026-06-20. 관점: **검색 품질 + 유니코드 정규화 + combobox 동작**.

## 점검 범위 / 방법

- 코드 Read: `searchCompanies` 매칭(`includes`)/정렬(`members` desc)/600 캡, API 라우트(q 50자 가드, try/catch), SearchBox 디바운스·AbortController·키보드·ARIA.
- 데이터 직접 분석: `data/companies.json` 을 Node 로 로드해 정렬 불변식·전각/NFD 매칭·사업자번호 분포를 **실측**(아래 수치는 측정값).
- 라이브 질의: dev 서버는 미가동(ECONNREFUSED)이라 HTTP 왕복은 미수행. 단, API 라우트는 `searchCompanies` 를 그대로 호출하므로 데이터층 실측이 종단 동작과 동일하다(라우트는 필드 매핑만 수행). 한글 쿼리를 curl 로 보내면 Git Bash 인코딩 때문에 깨지므로(round2-03 참조) Node 측정으로 대체했다.

## 발견 (심각도 / 근거)

### F-1. [중간] 유니코드 정규화 미처리 — NFD·전각·공백·중점 매칭 실패
- 근거: `searchCompanies` L150 은 `q.trim().toLowerCase()` 만 한다. `normalize()` 호출이 없고, `nameLc`(L146)도 `toLowerCase()` 만 전처리한다.
- **NFD(실측 결정타)**: 쿼리 "삼성"을 NFC 로 주면 1,981건, **NFD 자모분리형으로 주면 0건**. macOS 파인더/일부 IME·복붙은 NFD(자모분리) 문자열을 만든다. 데이터는 NFC(non-NFC 상호 단 1건: `주식회사임원（林元）`)인데 질의가 NFD 면 `includes` 가 전멸한다 — "검색이 안 된다"는 무결과로 보이지 사용자는 원인을 모른다.
- **전각 괄호(실측)**: `（주）`(전각) 상호 26,333건 중 **26,288건은 반각 `(주)` 쿼리로 도달 불가**. 사용자가 자연스럽게 반각 "(주)"를 입력하면 전각 표기 회사가 통째로 빠진다. 역방향도 동일.
- **공백·중점**: 공백 정규화가 없어 "주식회사 카카오"와 "주식회사카카오"가 갈린다(데이터에 둘 다 존재). 중점도 `·`/`ㆍ`/`・` 가 혼재(189건, 예: `대·중소기업·농어업협력재단`)라 어느 글자를 쳤는지에 따라 매칭이 갈린다.
- 개선안(P-1): 검색 키와 질의 양쪽에 동일 정규화 파이프라인 적용 — `s.normalize("NFC")` → 전각→반각(괄호·영숫자, `String.fromCharCode(c-0xFEE0)` 류) → 중점 통일(`[·ㆍ・]`→`·`) → `toLowerCase()` → 선택적으로 공백 제거. `nameLc` 전처리(L146)에서 한 번에 만들고 질의도 같은 함수를 거치게 해 비용을 모듈 로드 1회로 유지. 공백 제거 비교는 별도 정규화 컬럼을 둬 부분일치 의미를 보존하는 편이 안전.

### F-2. [양호 — 검증됨] 600 캡 + members desc 정렬 가정의 견고성
- 근거(실측): `A.name`/`A.members` 는 **members 내림차순으로 완전 정렬**(위반 0건). 스캔(L156 `for i=0..NA`)이 큰 회사부터 만나므로, L162 `hits.sort(members desc)` 후 상위10은 **항상 첫 600 히트 안에 포함**된다. 600 캡(L159 `if (hits.length > 600) break`)은 풀스캔만 절약하고 상위 결과를 누락시키지 않는다.
- **단, 이 보장은 전적으로 "배열이 members desc"라는 ETL 산출물 불변식에 의존**한다. 정렬이 깨진 채 재생성되면 캡이 즉시 정확도 버그로 바뀐다. 코드에 이 의존성을 명시한 주석/단언이 없다.
- 개선안(P-2): `searchCompanies` 상단에 "A 는 members desc 전제, 캡은 그 위에서만 정확" 주석. 가능하면 ETL 산출 시 정렬 단언, 또는 캡 직전 `hits` 가 이미 인덱스 오름차순(=members desc)임을 활용함을 명문화.

### F-3. [중간] 사업자번호 6자리 비고유 → 번호 검색 오탐
- 근거(실측): `bizNo` 는 전부 6자리, distinct 32,241개뿐(평균 ~17개사 공유, 최다 `114812`는 851개사 공유). 실제 10자리 사업자등록번호가 아니라 앞부분/부분 코드다.
- 영향: L154 `byDigit = digits.length >= 3` 로 3자리 이상 숫자면 번호 매칭(L157 OR `A.bizNo[i].includes(digits)`)이 켜진다. 6자리를 정확히 쳐도 수십~수백 회사가 동률 매칭돼 **특정 회사를 번호로 핀포인트할 수 없다**. SearchBox placeholder(L80 "회사명 또는 사업자번호 검색")의 약속과 어긋난다. 또 이름 속 숫자("123푸드")와 번호 매칭이 OR 로 섞여 의도 구분이 없다. 진짜 10자리 번호 입력 시 6자리 데이터엔 없어 거의 무결과.
- 개선안(P-3): placeholder 를 "상호명 검색"으로 수정하거나 진짜 10자리 번호를 데이터/검색에 도입. 최소한 6자리 정확매칭 시 "동일 코드 N개사" 안내, 또는 숫자 쿼리를 이름-숫자/번호로 분리.

### F-4. [중간] 접두/완전일치 가중 부재 — 랭킹이 members 단일 키
- 근거: L157 매칭은 `includes` 부분일치 불리언일 뿐, L162 정렬은 `A.members[b]-A.members[a]` 하나뿐. 매칭 위치(접두 vs 중간)·완전일치 여부·매칭 길이 비율을 보지 않는다.
- 영향: "소프트" 검색 시 이름 어디든 "소프트"가 든 가장 큰 회사가 1위(한국마이크로소프트)다. 사용자가 정확한 상호를 떠올려 입력해도 완전/접두 일치 회사가 더 큰 무관 회사 아래로 밀린다. 자동완성 UX 관례(접두 > 단어경계 > 부분일치 가중)가 없다. 완화: members 정렬이 "유명한 곳이 위"와 대체로 맞아 체감은 나쁘지 않으나 정확 질의에 약함.
- 개선안(P-4): `rank = (완전일치?3:접두?2:1)` 1차, members 2차의 2단 정렬. 비용은 히트 집합(≤600)에서만 계산하면 미미.

### F-5. [중간→낮음] SearchBox combobox 키보드·ARIA·상태 점검
- **양호한 부분**: `role="combobox"`+`aria-expanded`(L83, `open && results.length>0`로 정확)+`aria-controls`+`aria-autocomplete="list"`+`aria-activedescendant`(L86)로 패턴 골격은 갖췄다. `<ul role="listbox">`(L91), `<li role="option" id="search-opt-{i}" aria-selected>`(L93)도 정합. 옵션 버튼 `tabIndex={-1}`(L95)로 포커스가 입력에 머무는 것도 옳다. 디바운스 180ms + cleanup(L33) + **AbortController abort(L35)** 로 stale 응답을 막는 것도 잘 구현됨. `res.ok` 검사(L24)도 있다.
- **이슈 a — Escape 가 입력을 비우지 않음**: L50 Escape 는 `setOpen(false)` 만 한다(APG combobox 권장은 1차 목록 닫기, 2차 입력 클리어). 또 닫은 뒤 `active` 를 -1 로 리셋하지 않아 `aria-activedescendant` 가 닫힌 목록의 옛 옵션 id 를 가리킨 채 남을 수 있다 → 스크린리더가 존재하지 않는 활성옵션을 안내할 여지.
- **이슈 b — 무결과/에러 구분 불가**: catch(L29-31)는 AbortError 가 아니면 `setResults([])` 만 한다. 그러면 "검색 결과가 없습니다."(L117-121)가 **네트워크/500 에러에도 동일하게** 뜬다. 사용자는 "결과 없음"과 "검색 실패"를 구분 못 한다. 또 에러 시 `setOpen` 미조정으로 직전 드롭다운 상태가 어정쩡하게 남을 수 있다.
- **이슈 c — 무결과 박스에 ARIA 라이브/role 없음**: L117 무결과 `<div>` 는 단순 시각 텍스트라 스크린리더가 결과 변화를 능동 안내하지 않는다(`role="status"`/`aria-live="polite"` 부재). 결과 건수 안내(`aria-live`)도 없어 비시각 사용자는 "몇 건 떴는지" 알기 어렵다.
- **이슈 d — 마우스 hover 가 active 를 덮음**: L96 `onMouseEnter={()=>setActive(i)}`. 키보드로 이동 중 포인터가 목록 위에 있으면 hover 가 `active` 를 가로채 `aria-activedescendant` 가 튄다(키보드/마우스 동시 사용 시 혼란). 기능 버그는 아님.
- 개선안(P-5): Escape 2단계(목록 닫기→입력 클리어)+닫을 때 `setActive(-1)`; 에러 상태(`error` state) 분리해 "검색 중 오류" 별도 표기; 무결과/결과수에 `role="status" aria-live="polite"` 라이브 영역 추가; ArrowDown/Up 시 활성 옵션으로 스크롤 보장(긴 목록 대비).

## 개선 제안 (우선순위)

- **P-1 [F-1]** 검색 키·질의 공통 정규화: `normalize("NFC")` → 전각→반각 → 중점 통일 → `toLowerCase()`(+선택적 공백 제거 컬럼). `nameLc` 전처리에서 1회 생성. *(NFD 0건/전각 26,288건 누락이 실측된 가장 영향 큰 결함)*
- **P-2 [F-2]** members desc 의존성 주석/단언으로 600 캡 정확성 명문화.
- **P-3 [F-3]** placeholder 를 "상호명"으로 수정 또는 10자리 번호 도입; 6자리 동률 다수 사용자 안내.
- **P-4 [F-4]** 접두/완전일치 1차 가중 + members 2차의 2단 랭킹.
- **P-5 [F-5]** Escape 2단계 + `active` 리셋, 에러/무결과 상태 분리, `aria-live` 결과 안내.

## 종합

검색 **핵심 기능과 combobox 골격은 양호**하다 — 한글 부분일치 정상, 대형사 상위 노출 정상, 600 캡은 members-desc 정렬(위반 0건 실측) 덕에 결과를 누락시키지 않고, SearchBox 는 AbortController·`res.ok`·ARIA combobox 패턴을 갖췄다. **개선 핵심은 정규화**다: NFD 질의는 0건 매칭, 전각 `（주）`만 가진 26,288개사는 반각 입력으로 도달 불가 — macOS·복붙 사용자에게 "검색이 안 되는" 무결과로 나타난다. 그 외 사업자번호 6자리 비고유(핀포인트 불가), 접두/완전일치 가중 부재, combobox 의 Escape 2단계·에러/무결과 구분·`aria-live` 부재가 남는다.
