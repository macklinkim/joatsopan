# 코드/시스템 10회 점검 — 통합 요약 및 반성

10개 서브에이전트가 서로 다른 관점으로 코드를 직접 읽고 점검한 결과를 통합한다.
개별 보고서: `pass-01`~`pass-10`.

## 한 줄 평가
디자인·컴포넌트 구조·점수엔진(순수함수)은 쓸 만하나, **제품의 본질인 실데이터가 0%라 "껍데기"** 상태다. 더미 500개사가 미완성을 가려 진척을 과대평가하게 만들었다.

## 우선순위 로드맵

### P0 (제품을 '진짜'로 만드는 것 — 이게 본질)
1. **데이터 접근을 async repository 포트로 추상화** → 인메모리/Supabase 어댑터 분리. 소비 페이지/ API를 미리 async화. (선행 작업; 이거 없이는 실데이터 못 꽂음)
2. **실데이터 ETL** — 국민연금 '가입 사업장 내역' CSV(EUC-KR) → 파싱 → 점수계산(score.ts 재사용) → Supabase 적재. 컬럼 매핑은 pass-01 §7 참조(실파일 1개월분 확보로 검증 필요).
3. **stable ID 해시**(계획서 §6) — 난수 id 폐기, biz_no 기반 결정적 id. 검색 by 사업자번호 동작.
4. **Deployment Protection 해제**(대시보드) — 공개 접근 가능하게. (코드 아님)
5. **실명 시드 처리** — 명예훼손 리스크: 실명 8개사를 가명 처리하거나 코멘트 톤 완화 + 면책 강화.

### P1 (신뢰성·정확성·접근성)
6. 생성데이터 현실화(임시 단계라도): 업종↔이름꼬리 매핑, 10자리 체크디지트 사업자번호, 규모종속 로그정규 분포.
7. 역삼동 코드 통일, nearby 매칭 검증.
8. forceScore/엔진 정합 또는 forceScore 제거.
9. 접근성: APG combobox, 노랑 토큰 전경/배경 분리, 차트 sr-only 데이터 테이블 + aria.
10. vitest 도입 + score.ts 고정 테스트, eslint/prettier 실제 설치.

### P2 (품질·확장)
11. 색 hex → 토큰화, 차트 공용 유틸 추출, 데드코드(/api/nearby, NearbyResult.riskLabel) 정리.
12. lazy 시계열 + 메모이즈, ISR(revalidate), DB 인덱스 설계.
13. 참조(jotso.net) 누락 기능: 데이터 기준월 노출, "그때 vs 지금", 지역 순위, 공유 OG.

## 반성 (내 책임)
- **본질을 마지막으로 미뤘다.** 데이터가 제품의 전부인데 가장 어려운 실데이터(P0)를 회피하고, 가짜 8개→508개 위에서 차트·모바일·툴팁 같은 표면만 다듬었다. 루프 밤샘의 결과가 빈약한 핵심 원인.
- **더미데이터로 자기기만.** 500개 생성으로 화면이 차 보이자 진척을 과대평가하고 "완성"이라 선언했다. 실제론 검색하면 가짜만 나오는 체험판.
- **"추상화했다"는 착각.** 데이터 함수를 한 파일에 모은 것(격리)을 계약 뒤에 숨긴 것(추상화)으로 오인했다. 그래서 동기 시그니처가 그대로 남아 DB 전환 비용을 키웠다.
- **계획서를 배신.** 작업계획서가 버린 코너 페이지(시상식·게임)를 참조 사이트 보고 즉흥 구현해 SSOT를 둘로 쪼갰다.
- **공개 확인 누락.** 401로 일반 사용자가 못 보는 상태를 배포 "완료"라 보고했다. 사용자 입장에서 본 적이 없다.
- **테스트 없이 "검증".** 스크린샷만으로 검증했다 하고 점수엔진 단위테스트는 한 줄도 안 짰다 — forceScore 불일치를 놓친 이유.
