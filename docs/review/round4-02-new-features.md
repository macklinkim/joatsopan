## 발견

### Medium

**M1. `riskLadder`의 "더 위험한 곳 ↑ / 덜 위험한 곳 ↓"가 정수 점수 동점 탓에 약 92% 케이스에서 본인과 동일 점수를 위/아래로 표시 — "더/덜 위험"이라는 라벨이 사실상 거짓.**
`lib/data.ts:310` 정렬 `A.score[b]-A.score[a] || a-b`(점수 동점은 인덱스), `moreRisky=group.slice(pos-n,pos)`, `lessRisky=group.slice(pos+1,pos+1+n)`. 점수는 0~100 정수이고 분포가 극단적으로 뭉친다(실측 score=16이 218,226곳, =19가 27,253, =17이 26,867 …상위 6개 점수가 활성의 60%+). 그래서 같은 시군구 안에서도 동점 군집이 거대하다.
- 실측 1: 큰 시군구(2,935곳)의 중앙 회사 self=20 → `moreRisky=[20,20,20]`, `lessRisky=[20,20,20]`. 즉 위·아래 6곳 전부 **본인과 같은 20점**인데 UI는 위 3곳을 "↑ 더 위험한 곳", 아래 3곳을 "↓ 덜 위험한 곳"으로 표기.
- 실측 2(정량): 무작위 시군구·무작위 위치 2,990건에서 `moreRisky` 이웃의 **92.3%**, `lessRisky` 이웃의 **92.5%**가 self와 **동일 점수**. 즉 사다리에 뜨는 양옆 회사 10곳 중 9곳이 실제로는 "더/덜" 위험하지 않고 동급이며, 위/아래 배치는 오로지 인덱스 tie-break(데이터 적재 순서)로 갈린다.
영향: 핵심 기능(이직 판단 보조)의 의미가 무너짐 — "더 위험한 곳"에 본인과 똑같은 위험도 회사가 올라오고, 같은 점수의 두 회사가 서로의 상세를 열면 위/아래가 뒤바뀐다(A가 B를 "더 위험"으로, B가 A를 "더 위험"으로 표시 가능). 권고: 사다리 이웃을 **본인과 점수가 다른** 회사로 한정(`score>myScore`만 moreRisky, `<myScore`만 lessRisky)하거나, 동점은 "비슷한 위험도" 별도 그룹으로 분리. 최소한 동점 이웃에는 "동급" 표기를 붙일 것. (UI 가드 `moreRisky.length>0||lessRisky.length>0` 및 `group.length<4 → null`은 정상 작동하나 동점 문제는 가드와 무관.)

### Low

**L1. "갈아탈 만한 곳"(better)은 nearby 상위10(연봉순) 풀에서만 필터 — 상위10 밖의 "더 주면서 덜 위험"한 후보를 못 본다(누락이나 안전 측, 과대광고 아님).**
`app/company/[id]/page.tsx:48-52`. `nearbyResult = nearbyCompanies(id, 10)`은 같은 법정동→시군구→업종 중 첫 비공집합 tier를 **연봉 내림차순 top10**으로 자른다(`lib/data.ts:207-208`). `better`는 그 10개에서 `cur_salary > c.cur_salary && risk_score <= c.risk_score`를 만족하는 상위 3개.
- 필터 로직 자체는 **의도대로 정확**: "더 줌"(`>`, 동률 제외) + "위험 같거나 낮음"(`<=`, 본인 점수 이하 허용). 본인은 nearby 풀에서 이미 제외(`j !== g0`, `lib/data.ts:199`)되어 자기 자신이 추천에 끼지 않음. 표시 "{n.cur_salary/c.cur_salary}배"도 분모가 본인이라 항상 >1.0배로 정합.
- 한계: nearby가 연봉 top10이므로, 연봉이 본인보다 높은 회사가 그 tier에 10개 초과로 존재하면 **top10에 든 가장 높은 연봉대**만 후보가 된다. better의 salary 필터(`> c.cur_salary`)는 어차피 "더 높은 연봉"을 요구하므로 연봉순 top10은 항상 "가장 높은 연봉 후보들"을 포함 → **연봉 기준으로는 더 나은 후보를 놓치지 않는다**. 진짜 사각은 "연봉은 본인보다 높지만 top10에 못 들 정도(=top10 회사들보다 낮은 연봉)이면서 위험도가 매우 낮은" 회사 — 이런 곳은 better 후보에서 빠진다. 즉 "있는데 안 보임"(under-recommend)이지 "없는데 추천"(over-recommend)은 아니라 신뢰 측면 안전.
- 추가 엣지: tier 선택이 dong→sigungu→industry이므로, better 후보가 같은 동이 비면 같은 시군구, 그것도 비면 **업종(전국)** 풀에서 나올 수 있다. 이때 "갈아탈 만한 곳"이 전혀 다른 지역 회사가 될 수 있는데, 배너 카피는 "더 주면서 덜 위험"만 말하고 지역을 명시하지 않아 사용자가 같은 지역으로 오해할 여지(경미). 상위 `nearbyDesc`(61-66)는 scope를 정확히 문구화하지만 better 블록 자체엔 scope 표기가 없음.
권고(선택): better를 nearby 풀이 아니라 같은 시군구 전수에서 직접 산출하면 사각이 사라진다(성능은 시군구 그룹만 순회라 무해). 또는 현행 유지 + better 블록에 scope/지역 문구 추가.

**L2. `/about` 추정 공식·가중치·등급 임계가 `scoreCore.mjs` 실코드와 1:1 일치 — 신뢰 훼손 없음(핵심 확인 항목, 통과).**
`app/about/page.tsx:36-43` ↔ `lib/scoreCore.mjs:4-18` 대조:
- 직원 수: about "100인+ 0 / 30~99인 8 / 30인 미만 16" ↔ 코드 `members>=100?0:members>=30?8:16`. 경계 일치(99→8, 30→8, 29→16). ✔
- 추정 연봉: about "업종 중앙값 대비 낮을수록 최대 35점" ↔ 코드 `s2 = ratio>=1?0:round(35*min(1,1-ratio))`(`ratio=est_salary/median`). 최댓값 35·중앙값 이상 0점 일치. 공식 "당월고지금액÷가입자수÷0.09×12" ↔ `estSalary = round(noticeAmt/members/0.09*12/10000)`(about은 만원 단위 `/10000`만 생략 — 단위 표기로 허용 범위). ✔
- 회전율: about "(신규+상실)÷가입자수, 약 20%↑부터 가산해 최대 35점" ↔ 코드 `s3 = round(max(0,min(35,(turnover-20)*0.22)))`. 20%부터 가산·상한 35 일치. ✔
- 휴·폐업: about "상태 이상 시 30점" ↔ 코드 `s4 = is_closed?30:0`. ✔
- 등급: about "0~19 희귀 중소 · 20~49 보통 · 50+ 좋소 확정" ↔ `riskLabel`(14-18) `<20`/`<50`/else. 경계(19→희귀, 20→보통, 49→보통, 50→좋소) 일치. ✔
- 합산/클램프: about "4개 신호 합산 0~100" ↔ `max(0,min(100,s1+s2+s3+s4))`. ✔
틀린 가중치·임계 0건.

**L3. `SIDO_ALIAS` 시도명 통일이 레거시명을 100% 제거 — alias 자체가 만든 중복·부작용 0건.**
`scripts/etl.mjs:95-103` `parseRegion`이 첫 토큰을 `SIDO_ALIAS[t[0]] || t[0]`로 치환 후 인터닝. 실측:
- 적재된 sido 18개 목록에 레거시 `"강원도"·"전라북도"·"제주도"` **0건**(전부 `강원특별자치도`/`전북특별자치도`/`제주특별자치도`로 통일). ✔ (라운드3에서 레거시명이 별도 버킷으로 살아있던 문제가 해소됨.)
- alias가 같은 지역을 둘로 가르는 일 없음: 통일 후 sido 인터너에 동일 현행명이 단일 엔트리. 라운드3-05의 부수효과(빈/레거시 sido 회사가 탐색 필터로 도달 불가)도 레거시분은 해소.

**L4. 시도통일이 `regionRank`/`nearbyCompanies` 시군구 매칭에 부작용 없음 — 매칭은 `sidoIx+"|"+sgIx` 조합 키라 통일과 독립.**
`riskLadder`(308)·`regionRank`(264)·`nearbyCompanies`(202)는 모두 `mySido+"|"+mySg`로 그룹핑. alias는 같은 물리 지역을 **하나의** sidoIx로 모으므로 통일은 오히려 매칭 정합을 개선(강원도/강원특별자치도가 갈리지 않음). 실측 부작용 점검:
- 법정동(`bdong`) 단일 키 tier(nearby dong, `lib/data.ts:201`)가 **여러 sido에 걸치는** bdong 코드 5개 발견 → **전부 빈 sido(`""`) 레코드가 원인**(예: bdong `4159034023`이 `경기도:3`+`"":2`). 즉 잘못된/누락된 지번주소로 sido가 빈 레코드가 같은 bdong에 섞인 것이지 **alias와 무관**. 영향: nearby dong tier가 드물게 빈-sido 회사를 한두 곳 끼울 수 있으나(전국 빈-sido 활성 109곳·단일 시군구 그룹), 같은 bdong이면 실제 동일 법정동이라 추천 적절성은 대체로 유지. alias 통일이 만든 문제 아님.
- `riskLadder`/`regionRank`는 sido+sg 조합 키라 빈-sido끼리(`""|sg` 1그룹, 109곳)만 묶여 다른 정상 지역과 안 섞임. ✔

### Info

- `riskLadder` 가드 정상: `g>=NA → null`(휴폐업 제외, 306), `group.length<4 → null`(309), 본인 제외는 slice가 `pos`를 양쪽에서 배제(`slice(pos-n,pos)`·`slice(pos+1,…)`)로 자동 처리 — self가 결과에 안 들어옴. ✔ 끝단(최고/최저 위험) 시 빈 배열 → UI가 "이 지역 최고/최저 위험도입니다" 문구로 폴백(228, 241).
- `riskLadder`는 `bySg` 인덱스를 `slice()` 복사 후 정렬 → 캐시된 인덱스 배열 in-place 오염 없음(다른 함수가 같은 배열을 다른 정렬로 씀). ✔
- "갈아탈 만한 곳" 배너는 `better.length>0`일 때만 렌더(250) — 후보 없으면 미표시, 빈 블록 없음.
- `/about`의 한계 면책(48-50)이 "추정 연봉 상한 과소평가·추이 추정 곡선·사번 일부 표기"를 정확히 고지 — 데이터 한계와 코드 동작 일치.

## 권고 우선순위

1. **M1**: `riskLadder`를 본인과 **점수가 다른** 이웃으로 한정(`>myScore`/`<myScore`)하거나 동점을 "비슷한 위험도"로 분리 표기 — 현재 양옆의 ~92%가 동점인데 "더/덜 위험"으로 호도하는 것 해소. 신규 기능 중 정확성 영향 최대.
2. **L1**: "갈아탈 만한 곳"을 nearby top10 풀이 아니라 같은 시군구 전수에서 산출(사각 제거), 또는 better 블록에 scope/지역 문구 추가.
3. **L4 부수효과**: 빈-sido(`""`) 레코드는 alias 무관 별개 데이터 품질 이슈 — ETL에서 빈 sido 폴백/필터링 검토(선택).
