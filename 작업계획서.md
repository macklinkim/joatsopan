# 좋소판별기(jotso.net) 개인용 클론 — 작업계획서

> 목적: jotso.net이 대중화되며 사라질 것에 대비해, **개인용으로** 동일 기능을 직접 구축한다.
> 범위 한정: **회사 검색 + 중소기업(좋소) 판별** 기능에만 집중한다.
> 제외: 별이된좋소(폐업), 좋소시상식, 좋소게임, 이달의좋소 랭킹 등 부가 콘텐츠 전부 버린다.

> 📄 **기능·화면이 헷갈리면 [`Page_text_export.txt`](./Page_text_export.txt)를 먼저 보라.**
> 원본 jotso.net 회사 상세 페이지(주식회사 샤이닝라이언)를 그대로 추출한 **실제 예시**다. 헤더·핵심 지표 4종·기여 점수·한줄평·추천 등 "이 사이트가 실제로 무엇을 보여주는지"의 기준 샘플이며, 본 계획서의 지표/공식(2장)도 이 화면 값에서 역산했다.

## 0. 디자인 핵심 전략 (Design Strategy — 'Precision Insight')

> jotso.net의 핵심 기능을 분석하여 현대적이고 심플한 디자인으로 재해석한다. **구현 기준은 [`DESIGN.md`](./DESIGN.md)가 단일 기준(SSOT)**이며, 전체 명세는 → **12장 디자인 명세** 참조.
> ※ 아래 색/폰트는 `DESIGN.md` 확정 토큰을 따른다(초안의 `#FF4B4B`/`#2ECC71`/`Pretendard`는 `DESIGN.md` 값으로 대체됨).

**디자인 계획서 요약 (Design Specification)**
- **핵심 컨셉**: 데이터 중심의 명확한 정보 전달과 높은 반응성을 갖춘 `Precision Insight` 스타일(Corporate Minimalism). 지표가 장식보다 먼저 읽히는 paper-white 금융 터미널 감성.
- **컬러 전략**: 위험 지표 `risk-high #D8362A` ↔ 안정 지표 `risk-safe #2A8D5C`(+ 주의 `risk-warning #FEE500`)를 대조시켜 직관적인 정보를 제공한다. (색만으로 판단 강요 금지 — 아이콘·라벨 동반, 12.8 참조.)
- **컴포넌트**: 라운드값(Soft 8px)이 적용된 카드 레이아웃 + 데이터 시각화 차트를 중심으로 구성해 가독성을 높인다. 그림자 최소(톤 레이어·1px 저대비 보더).

**디자인 시스템 (Design System — `DESIGN.md`)**
- **타이포그래피**: **Hanken Grotesk**(헤드라인) + **Inter**(본문) + **JetBrains Mono**(숫자·데이터) 3-font로 데이터 밀도가 높은 화면에서도 선명한 가독성을 확보한다.
- **토큰 시스템**: 배경색(`surface-paper`)·서피스 색상(`surface-white`)·강조색(`risk-*`) 등을 의미 기반 토큰으로 정의해 일관된 디자인 출력을 보장한다.

## 1. 데이터 원천 (역공학 결론 — 확정)

원본 사이트 하단 고지문:
> "※ 본 결과는 **공공데이터(국민연금)** 기반 추정치이며 참고용입니다."

이 한 줄이 모든 것의 출처다. 실제 데이터셋은 다음 하나다.

- **출처**: 공공데이터포털(data.go.kr)
- **데이터셋**: `국민연금공단_국민연금 가입 사업장 내역` (월 단위 갱신, CSV 전월분 일괄 제공)
- **단위**: 사업장(직장) 1곳 = 1행, **매월 1개 스냅샷**

### 원본 CSV 주요 컬럼 → 사이트 지표 매핑

| 원본 컬럼 | 사이트에서 쓰는 곳 |
|---|---|
| 자료생성년월 | 시계열 X축 (예: 2015-11 ~ 2026-04) |
| 사업장명 | 회사명 |
| 사업자등록번호(앞 6자리) | 화면의 "사업자 119863" (공개 데이터는 **6자리만** 제공) |
| 업종코드 / 업종코드명 | "응용 소프트웨어 개발 및 공급업" 등 |
| 사업장 도로명/지번 주소 | "서울특별시 구로구 …", 지역 랭킹 |
| 법정동/행정동 주소코드 | 구/동 단위 비교("구로구 좋소 순위") |
| **가입자수** | **직원 수** (현원) |
| **당월고지금액** | **추정 평균연봉** 계산 입력값 |
| **신규취득자수** | **입사** |
| **상실가입자수** | **퇴사** |
| 사업장가입상태코드 / 탈퇴일자 | **휴·폐업 추정** |

> ⚠️ 핵심 함의: 공개 데이터는 사업자번호 6자리만 준다 → **사업장을 유일 식별하는 키가 원본에 없다.**
> jotso.net의 12자리 URL ID(`361104670471`)는 원본 값이 아니라 **사이트가 자체 생성한 내부 키**다.
> 우리도 `(사업장명 + 6자리 사번 + 주소코드)` 해시로 **자체 stable ID**를 만들어야 한다. (→ 6장)

## 2. 판별 기준(지표) 정의 — 역공학 완료

원본 화면에서 실제 값을 채취해 **공식을 그대로 복원**했다. 추정이 아니라 검산까지 끝낸 결과다.

### 2.1 직원 수 (현원)
- 정의: 최신 월의 `가입자수`.
- 밴드: `100명 이상`=중소 상단, `30~99명`=중소 중간, `<30명`=영세(추정).

### 2.2 입사 / 퇴사
- 입사 = 해당 월 `신규취득자수`, 퇴사 = 해당 월 `상실가입자수`.
- 시계열로 누적해 "입·퇴사 흐름" 차트와 근속 추정에 사용.

### 2.3 회전율 (turnover) — **공식 검산 완료**
```
회전율(%) = (당월 입사 + 당월 퇴사) / 현원 × 100
```
- 검산 ① 샤이닝라이언: 현원 49, 입사 47, 퇴사 47 → (47+47)/49 = 191.8% → 표기 **192%** ✓
- 검산 ② 엠피테크: 현원 10, 입사 9, 퇴사 8 → (9+8)/10 = **170%** ✓
- 월 퇴사율 = `퇴사 / 현원`(별도 표기, 예: "월 퇴사 1.0%").
- 라벨: 낮음=`안정적`, 높음=`사람이 자주 갈림`.

### 2.4 추정 평균연봉 — **계산식 복원**
국민연금 보험료는 기준소득월액의 **9%**. 따라서:
```
1인당 월 기준소득 = (당월고지금액 / 가입자수) / 0.09
추정 평균연봉      = 1인당 월 기준소득 × 12
```
- 단, 국민연금에는 **기준소득월액 상한(상한액)** 이 있어 고연봉은 과소추정된다 → "추정치" 고지가 붙는 이유.
- 비교 라벨은 **업종 중앙값** 대비 상대값:
  - `업종 중앙값 이상`, 또는 `업종 중앙값의 16%`처럼 비율로 표기.
  - → 우리는 **업종코드별 평균연봉 중앙값 테이블**을 미리 만들어야 한다.

### 2.5 휴·폐업 추정
- 신호: 최근 월 `가입자수`가 0으로 급락 / 사업장 탈퇴일자 존재 / 가입상태코드 변경.
- 라벨: `정상 영업` ↔ `휴폐업 신호`.
- (단, "별이된좋소" 화면 자체는 범위에서 제외 — 점수 가산 신호로만 사용)

### 2.6 위험도(좋소 점수) — **합산식 + 가중치 검산 완료**

화면의 "핵심 지표" 4개 옆 기여 점수를 모두 채취해 합이 총점과 일치함을 확인:

| 회사 | 직원수 | 연봉 | 회전율 | 휴폐업 | **합=위험도** |
|---|---|---|---|---|---|
| 샤이닝라이언 | +8 | +35 | +35 | 0 | **78** ✓ |
| 맑은소프트 | +8 | 0 | 0 | 0 | **8** ✓ |
| 아나패스(131명) | 0 | 0 | 0 | 0 | **0** ✓ |

```
위험도 = clamp(0, 100, 직원수점수 + 연봉점수 + 회전율점수 + 휴폐업점수)
```
복원된 점수 규칙(가중치는 위 관측으로 보정):
- **직원수점수**: 100명+ → 0, 30~99명 → +8, 30명 미만 → 더 큼(영세 가산). *큰 회사일수록 안전.*
- **연봉점수 (상한 ≈ 35)**: 업종 중앙값 이상 → 0. 중앙값 대비 비율이 낮을수록 선형 가산
  (중앙값의 16% → +35). `점수 ≈ 35 × (1 − 비율)` 형태로 캘리브레이션.
- **회전율점수 (상한 ≈ 35)**: 안정적 → 0, 회전율이 높을수록 선형 가산(192% → +35).
- **휴폐업점수**: 정상 0, 신호 있으면 큰 가산.

> 등급 라벨(위험도 → 문구): 낮음 = `희귀 중소`("사람이 안 나가는 회사네요"),
> 높음 = `좋소 확정`. 경계값은 우리가 정해도 무방(예: 0–20 희귀 / 20–50 보통 / 50+ 좋소).

**→ "이 회사가 좋은지 나쁜지"의 판단은 결국 이 4개 신호다: 회전율↑·연봉↓·영세·휴폐업신호 = 나쁨.**

## 3. 기술 스택 (원본과 동일 노선 권장)

원본은 **Next.js(App Router)** + 서버 렌더링. 회사 데이터는 페이지에 SSR로 박혀 있고, 검색만 `/api/search` XHR.

| 영역 | 선택 | 이유 |
|---|---|---|
| 프레임워크 | Next.js (App Router) | 원본과 동일, SSR로 회사 페이지 OG/SEO까지 그대로 |
| **배포** | **Vercel (외부 공개)** | 외부에서 접근 가능해야 함. Next.js 네이티브 호스팅 |
| **DB** | **Supabase (Postgres)** | 외부 공개 배포에는 로컬 SQLite 파일이 부적합(서버리스·임시 파일시스템). 남는 Supabase 인스턴스 활용 |
| 적재(ETL) | Node 스크립트 (월 1회 배치) | CSV → Supabase upsert. 로컬 또는 GitHub Action에서 실행 |
| DB 접속 | `supabase-js` / Postgres 풀러 | **서버리스 함수는 Supabase Connection Pooler(6543, transaction mode) 사용** — 커넥션 고갈 방지 |
| 검색 | Postgres `ILIKE` + `pg_trgm`(또는 tsvector) | 회사명 부분일치. 인덱스로 가속 |
| 차트 | 서버 생성 SVG | 원본도 이미지(SVG) 방식 |
| 권한 | RLS: 공개 테이블 **읽기 전용(anon select)** | 외부 공개지만 쓰기는 ETL(service_role)만 |

> 참고: 로컬 개발은 SQLite로도 가능하나, **배포 DB는 Supabase로 단일화**한다(스키마는 5장 Postgres 버전).

## 4. 시스템 아키텍처

```
[공공데이터포털 CSV(월)] 
        │  (1) 다운로드
        ▼
[ETL 배치 스크립트] ── 파싱/정규화/자체ID생성/지표·점수 사전계산
        │  (2) upsert (service_role)
        ▼
[Supabase Postgres: companies / monthly_stats / industry_median]
        │   (Vercel 서버리스 → Connection Pooler 경유)
        ├─(3a) /api/search?q=     ──▶ 검색 자동완성
        ├─(3b) /company/[id]      ──▶ SSR 상세(지표+차트+점수)
        └─(3c) /api/nearby?id=    ──▶ 주변 회사 추천(동 단위·연봉순)
```

핵심: **점수·회전율·추정연봉은 조회 때 계산하지 말고 ETL 단계에서 미리 계산해 저장**한다(개인용이라 단순·빠름).

## 5. 데이터 모델 (Supabase Postgres 스키마)

> 배포 DB는 Supabase(Postgres). 아래는 Postgres 표기. (로컬 SQLite 개발 시 `TEXT/INTEGER`로 치환 가능.)

```sql
-- 회사 기본 정보 (최신 스냅샷 요약 포함)
CREATE TABLE companies (
  id            TEXT PRIMARY KEY,   -- 자체 생성 stable ID (6장)
  biz_name      TEXT NOT NULL,
  biz_no6       TEXT,               -- 사업자번호 앞 6자리
  industry_code TEXT,
  industry_name TEXT,
  sido          TEXT,               -- 광역시도
  sigungu       TEXT,               -- 구/군 (예: 구로구)
  bdong_code    TEXT,               -- 법정동 코드 (동 단위 '주변 추천' 매칭 키, 8.3.6)
  dong          TEXT,               -- 법정동/행정동명 (예: 구로동)
  addr          TEXT,
  -- 최신월 요약 (검색·목록·점수용, ETL에서 채움)
  cur_members   INTEGER,           -- 현원
  cur_salary    INTEGER,           -- 추정 평균연봉(만원)
  cur_turnover  REAL,              -- 회전율 %
  risk_score    INTEGER,           -- 위험도 0~100
  risk_label    TEXT,              -- 희귀중소/보통/좋소확정
  is_closed     INTEGER DEFAULT 0, -- 휴폐업 추정
  last_ym       TEXT               -- 최신 자료년월
);
CREATE INDEX idx_companies_name ON companies(biz_name);
CREATE INDEX idx_companies_region ON companies(sido, sigungu, risk_score);
CREATE INDEX idx_companies_industry ON companies(industry_code, cur_salary);
-- 주변 추천: 동 단위 + 5인 초과 + 연봉순 정렬 가속
CREATE INDEX idx_companies_nearby ON companies(bdong_code, cur_salary DESC) WHERE cur_members > 5;

-- 월별 시계열 (차트용)
CREATE TABLE monthly_stats (
  company_id  TEXT NOT NULL,
  ym          TEXT NOT NULL,      -- 'YYYY-MM'
  members     INTEGER,            -- 가입자수=직원수
  hires       INTEGER,            -- 신규취득=입사
  leaves      INTEGER,            -- 상실=퇴사
  notice_amt  INTEGER,            -- 당월고지금액
  est_salary  INTEGER,            -- 추정연봉(만원)
  turnover    REAL,               -- 회전율 %
  PRIMARY KEY (company_id, ym)
);

-- 업종별 연봉 중앙값 (연봉 점수 계산용, ETL에서 매월 재계산)
CREATE TABLE industry_median (
  industry_code TEXT PRIMARY KEY,
  median_salary INTEGER           -- 만원
);
```

## 6. 자체 식별자(ID) 생성 규칙

공개 데이터에 고유키가 없으므로:
```
id = base36( hash( biz_name_정규화 + '|' + biz_no6 + '|' + 법정동주소코드 ) )
```
- 회사명 정규화: 공백/「(주)·주식회사」 제거 후 비교(동명이인 사업장 구분 위해 주소코드 포함).
- 월이 바뀌어도 같은 사업장은 같은 ID가 나와야 시계열이 이어진다 → 입력은 **변하지 않는 필드만** 사용.

## 7. ETL 배치 (월 1회) — 처리 순서

1. **다운로드**: 공공데이터포털에서 최신월 CSV 받기(수동 또는 API).
2. **인코딩/파싱**: EUC-KR 가능성 → UTF-8 변환, 컬럼 매핑.
3. **정규화 & ID**: 6장 규칙으로 `company_id` 생성. **주소에서 `bdong_code`(법정동코드)·`dong`(동명) 추출 → companies에 저장**(주변 추천용, 8.2b).
4. **monthly_stats upsert**: 회전율·추정연봉 계산해 함께 저장.
   - `turnover = (hires+leaves)/members*100`
   - `est_salary = round((notice_amt/members)/0.09*12 / 10000)` (만원)
5. **업종 중앙값 재계산**: 업종코드별 `est_salary` 중앙값 → `industry_median`.
6. **위험도 계산 & companies 갱신** (2.6 공식):
   - 직원수점수 + 연봉점수(업종중앙값 대비) + 회전율점수 + 휴폐업점수 → clamp(0,100).
   - risk_label 매핑.
7. **휴폐업 판정**: 직전월 존재 → 이번월 `members=0`/미등장 → `is_closed=1`.

> 개인용이므로 1~2개월치만 있어도 동작. 시계열 차트는 누적된 만큼만 그린다.

## 8. 화면/기능 (범위 내 — 이것만 만든다)

### 8.1 홈 `/` — 검색 only
- 중앙 검색창(자동완성) **하나만** 둔다.
- 원본 홈의 랭킹·폐업·시상식 섹션은 **전부 제거**.

### 8.2 검색 API `/api/search?q=`
- 응답(원본과 동일 형태):
```json
{ "results": [ { "bizNo":"…", "bizName":"…", "industry":"…", "members": 256 } ] }
```
- 회사명 부분일치, `members` 내림차순 정렬, 상위 8~10개.

### 8.2b 주변 추천 API `/api/nearby?id=`
- 기준 회사의 `bdong_code`(법정동) 기준 같은 동 회사 추천. 동 결과가 적으면 `sigungu`로 폴백.
- **필터**: `cur_members > 5`(5인 이하 제외) AND `is_closed = 0` AND `id != 기준회사`.
- **정렬**: `cur_salary DESC`(연봉순), 상위 10개.
- 응답 예:
```json
{ "results": [ { "id":"…", "bizName":"…", "salary": 4200, "members": 48, "riskScore": 12 } ] }
```
```sql
-- 핵심 쿼리
SELECT id, biz_name, cur_salary, cur_members, risk_score
FROM companies
WHERE bdong_code = $1 AND cur_members > 5 AND is_closed = 0 AND id <> $2
ORDER BY cur_salary DESC NULLS LAST
LIMIT 10;
```

### 8.3 회사 상세 `/company/[id]` — 판별 결과 (핵심 화면)
원본에서 **남길 블록만**:
1. **헤더**: 회사명 / 업종 / 사업자6자리 / 주소 / **위험도 N/100 + 등급 라벨 + 한줄평**.
2. **핵심 지표 4종**(각 기여 점수 표기):
   - 직원 수 · 추정 평균연봉 · 회전율 · 휴폐업 여부.
3. **직원 수 추이** 차트(SVG, 월별 members).
4. **평균 연봉 추이** 차트(월별 est_salary).
5. **입·퇴사 흐름** 차트 + 근속/월퇴사율.
6. **주변 회사 추천**(원본 기능 재현 — `/api/nearby`):
   - 원본 jotso.net은 검색된 회사의 **위치(동 단위)** 기준으로 주변 회사를 추천한다(예: 구로동 회사면 같은 구로동 회사들).
   - 매칭 키: **`bdong_code`(법정동) 동일** → 결과 부족 시 같은 `sigungu`로 폴백.
   - 정렬: **추정 평균연봉(`cur_salary`) 내림차순** = "주변에서 더 주는 곳".
   - **제외 규칙: 5인 이하 사업장(`cur_members <= 5`) 제외** — 추가 채용 가능성이 낮아 이직 추천 가치가 낮음. (휴폐업·본인 회사도 제외.)
   - 상위 N개(예: 10개) 카드 리스트. 각 카드: 회사명 · 추정연봉 · 직원수 · 위험도 배지.
   - (보조) "갈아탈 만한 곳" = 동일 동/구 `risk_score` 낮은 순 — 여력 되면 탭으로 함께.

### 8.4 버리는 것 (구현 안 함)
- ❌ 별이된좋소(`/graveyard`) · 좋소시상식(`/awards`) · 좋소게임(`/games`) · 이달의좋소 랭킹(`/rankings`)
- ❌ 카카오 공유 / OG 카드 이미지 생성 / 월드컵 / "관짝 열고 나온 회사" 등 밈 콘텐츠
- (휴폐업 "신호"는 점수 입력으로만 쓰고, 전용 페이지는 안 만든다)

## 9. 점수 엔진 모듈 (의사코드)

```js
function riskScore(c, industryMedian) {
  // 1) 직원수 (작을수록 위험)
  let s1 = c.members >= 100 ? 0 : c.members >= 30 ? 8 : 16;

  // 2) 연봉 (업종 중앙값 대비 낮을수록 위험, 상한 35)
  const ratio = industryMedian ? c.est_salary / industryMedian : 1;
  let s2 = ratio >= 1 ? 0 : Math.round(35 * Math.min(1, 1 - ratio));

  // 3) 회전율 (높을수록 위험, 상한 35) — 약 60%↑ 부터 가산되게 보정
  let s3 = Math.round(Math.max(0, Math.min(35, (c.turnover - 20) * 0.22)));

  // 4) 휴폐업 신호
  let s4 = c.is_closed ? 30 : 0;

  return Math.max(0, Math.min(100, s1 + s2 + s3 + s4));
}
```
> 계수(0.22, 시작점 20% 등)는 채취한 샘플(192%→+35, 2%→0)에 맞춰 1차 보정한 값.
> 회사 몇 곳을 더 대입해 미세조정하면 원본과 거의 일치시킬 수 있다.
